# 技术选型报告 v2

## 1. 选型目标

本选型不以“实现最快”或“上手最容易”为目标，而以以下指标排序：

1. 长期桌面交互性能。
2. 训练生态兼容性。
3. 架构扩展能力。
4. 长时间运行稳定性。
5. 开源产品质感。

## 2. 候选方案

### 2.1 方案 A：纯 Python + PySide6

优点：

1. 与 Ultralytics 生态一致。
2. 全链路开发速度快。
3. 原型实现门槛低。
4. 打包难度相对可控。

缺点：

1. 标注画布极限性能不足。
2. 几何编辑内核长期扩展性一般。
3. 大规模图像交互场景中原生质感和资源控制能力较弱。

结论：

1. 适合作为原型路线。
2. 不作为终局路线。

### 2.2 方案 B：纯 C++ + Qt

优点：

1. 原生桌面性能强。
2. 标注引擎与渲染引擎能力强。
3. 长期维护桌面核心体验更优。

缺点：

1. YOLO 训练生态不在 C++ 主战场。
2. 模型兼容、导出、脚本工具集成成本更高。
3. 未来更换训练后端时维护复杂。

结论：

1. 适合作为桌面核心。
2. 不适合作为唯一语言栈。

### 2.3 方案 C：Electron 或 Tauri + Python

优点：

1. UI 开发表现力强。
2. 前端生态丰富。

缺点：

1. 并非工业图像标注工作台的最优形态。
2. 资源占用与交互性能不如 Qt 原生方案。
3. 技术栈会被拆成更多层次，治理复杂。

结论：

1. 不推荐作为主方案。

### 2.4 方案 D：Qt/QML + C++ + Python 后端混合架构

优点：

1. 兼顾桌面性能与训练生态。
2. 标注与训练职责边界清晰。
3. 可把高性能几何编辑放在 C++，把模型生态放在 Python。
4. 适合长期演进为成熟工业产品。

缺点：

1. 文档、构建和联调复杂度更高。
2. IPC 契约设计必须严格。

结论：

1. 这是最终推荐方案。

## 3. 最终定稿

### 3.1 前端与桌面核心

1. C++17 或以上。
2. Qt 6。
3. QML 用于页面与组件层。
4. C++ Widgets 仅用于必要的底层控制或复杂嵌入场景。

### 3.2 训练与推理后端

1. Python 3.11+
2. Ultralytics 作为第一训练后端。
3. ONNX Runtime 用于导出后验证。

### 3.3 元数据与资产层

1. SQLite 作为本地元数据数据库。
2. 文件系统作为原始数据和模型资产仓库。

### 3.4 进程通信

建议采用本地 JSON-RPC 风格 IPC 或结构化消息协议。

要求：

1. 消息结构固定。
2. 支持请求、响应、事件通知三类消息。
3. 支持长任务日志流。

## 4. 技术选型落地约束

1. UI 不得直接调用 Python 训练逻辑。
2. 所有训练、推理、导出都必须通过后端服务接口完成。
3. C++ 层必须只处理桌面核心与高性能图像交互。
4. Python 层必须避免承担复杂 UI 状态管理。
