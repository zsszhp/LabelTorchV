# 标炬（LabelTorch）产品宪章 v2

## 1. 产品定义

标炬不是单纯的标注软件，也不是单纯的 YOLO 训练前端，而是一套面向工业缺陷检测场景的本地离线视觉数据治理与模型闭环平台。

平台职责包括：

1. 管理项目、数据集、类别体系、标签修订历史。
2. 提供高性能本地图像浏览与标注修订能力。
3. 管理检测模型训练、评估、版本、导出与复现。
4. 提供历史模型辅助标注与增量训练闭环。
5. 为后续 OBB、分类、异常检测、主动学习和多模型适配提供稳定底座。

## 2. 目标用户

### 2.1 当前核心用户

1. 单人使用的工业算法工程师。
2. 主要工作平台为 Windows。
3. 主要使用 NVIDIA GPU 执行训练。
4. 当前数据集以 YOLO txt 为主。

### 2.2 未来扩展用户

1. 在 Ubuntu 上进行模型训练和大规模评估的算法工程师。
2. 需要长期管理多项目、多类别体系、多版本数据与模型的高级用户。

## 3. 产品目标排序

本项目所有设计、取舍和实现均遵循以下优先级：

1. 训练效果最优。
2. 标注与审核效率最优。
3. 稳定性与可恢复性最优。
4. 本地交互性能最优。
5. 长期架构可扩展性。
6. 开发效率与实现成本。

说明：当“更快做完”和“最终质量更高”产生冲突时，统一选择最终质量更高的方案。

## 4. 产品原则

### 4.1 离线优先

1. 核心功能不依赖云端服务。
2. 数据、训练、版本、导出均可在断网环境下运行。

### 4.2 数据治理优先

1. 不允许只管理模型版本而忽略数据版本。
2. 所有训练结果必须可追溯到明确的数据快照与配置快照。

### 4.3 标注修订优先于“纯查看”

1. 辅助标注不是只读能力。
2. 用户必须能够直接修正错误候选框、补框、删框、改类。

### 4.4 训练后端可替换

1. 首发版本采用 Ultralytics 体系。
2. 平台内部不得将业务流程与单一训练后端强耦合。

### 4.5 不可逆操作必须可审计

1. 类别映射重写。
2. 标签写回。
3. 数据划分重建。
4. 模型导出覆盖。

上述操作必须记录完整审计信息，必要时必须具备回滚点。

## 5. 终局愿景

成熟后的标炬应具备以下能力：

1. 作为工业视觉项目的单机工作台使用。
2. 覆盖数据导入、标注修订、训练、评估、导出、发布全链路。
3. 在 100 到 10000 张图像规模下稳定运行。
4. 在检测任务成熟后可继续扩展到 OBB、分类与异常检测。
5. 具备面向开源社区展示的项目质量与发布规范。

## 6. 明确非目标

在当前产品路线中，下列方向不是近期主目标：

1. 云端协作平台。
2. 多用户权限系统。
3. 浏览器优先的 Web 端体验。
4. 与大规模标注团队协作的任务派发系统。
5. SaaS 化服务与在线推理托管。
