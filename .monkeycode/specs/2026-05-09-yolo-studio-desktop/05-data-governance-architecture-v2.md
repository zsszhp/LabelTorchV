# 数据治理架构 v2

## 1. 目标

将“数据版本治理”提升到与“模型版本治理”同等重要的层级，确保任何训练结果都可追溯到明确数据状态。

## 2. 数据治理范围

1. 原始图片资产。
2. 原始标签资产。
3. 标签修订版本。
4. 类别体系与映射版本。
5. 数据划分结果。
6. 训练快照。

## 3. 项目级类别体系

项目必须存在标准类别体系 `Project Taxonomy`。

作用：

1. 统一多批次导入数据的类别语义。
2. 统一后续训练、评估和导出使用的类别顺序。

## 4. 导入级类别体系

每次导入的原始标签类别都必须单独记录。

原因：

1. 不同来源数据可能类别顺序不同。
2. 某些数据集只有 class id，无语义名称。

## 5. 类别映射重写策略

### 5.1 原则

1. 映射必须显式确认。
2. 映射必须版本化。
3. 映射重写前必须预览影响范围。

### 5.2 重写流程

1. 抽取导入类别。
2. 用户建立映射。
3. 系统预览变更统计。
4. 用户确认。
5. 系统生成 ClassMappingRevision。
6. 系统执行标签重写并生成变更日志。

## 6. 标签修订治理

### 6.1 修订来源

1. 人工标注编辑。
2. 模型辅助标注确认。
3. 类别映射重写。

### 6.2 修订记录

每次写回必须记录：

1. 修改前内容。
2. 修改后内容。
3. 修改原因或来源。
4. 修改时间。

## 7. 数据快照

### 7.1 定义

数据快照表示用于训练的稳定输入集合。

### 7.2 快照内容

1. 样本清单。
2. 类别体系版本。
3. 标签修订边界。
4. 训练/验证/测试划分。

### 7.3 原则

1. 一次训练只能绑定一个快照。
2. 快照不可变。
3. 新增样本或修改标签后必须生成新快照。

## 8. 数据治理成熟要求

成熟版本必须支持：

1. 快照对比。
2. 修订影响分析。
3. 错误样本回流统计。
4. 数据质量分析。
