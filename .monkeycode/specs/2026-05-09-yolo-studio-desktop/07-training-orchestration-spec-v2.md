# 训练编排规范 v2

## 1. 目标

训练编排系统负责将数据快照、训练配置、运行环境、日志、指标和模型产物组织成稳定、可恢复、可追溯的训练流程。

## 2. 训练抽象层

### 2.1 统一训练接口

平台内部定义 `TrainingAdapter` 抽象，不允许业务层直接依赖 Ultralytics 实现细节。

### 2.2 必需接口

1. `validate_config`
2. `prepare_dataset_snapshot`
3. `start_training`
4. `stop_training`
5. `parse_logs`
6. `collect_metrics`
7. `export_model`

### 2.3 首发适配器

1. `UltralyticsAdapter`

## 3. 模型家族支持策略

MVP 必须支持：

1. `yolov5`
2. `yolov8`
3. `yolov8_obb`
4. `yolov10`
5. `yolov11`

要求：

1. UI 层以统一模型家族视图展示。
2. 差异化参数由适配器内部消化。

## 4. 训练任务生命周期

状态定义：

1. `draft`
2. `validated`
3. `queued`
4. `running`
5. `stopping`
6. `succeeded`
7. `failed`
8. `cancelled`

状态规则：

1. 未通过配置校验不得进入 `queued`。
2. `running` 必须绑定日志流与心跳。
3. `failed` 必须记录失败原因与原始日志路径。

## 5. 配置快照要求

每次训练必须固化：

1. 数据快照 id。
2. 模型家族和变体。
3. 所有用户输入参数。
4. 推导出的默认参数。
5. 运行环境信息。
6. 父模型版本信息。

## 6. 早停策略

1. 默认 `patience=50`。
2. 默认参考主要验证指标为 `mAP50-95`。
3. 必须允许用户修改 `patience`。
4. UI 必须明确显示早停是否触发及触发时机。

## 7. 增量训练策略

1. 新训练任务可选择父模型版本作为初始化权重。
2. UI 必须清晰区分：从零训练、预训练权重训练、增量训练。
3. 训练完成后模型版本必须记录父版本关系。

## 8. 后续增强路线

### 8.1 V2

1. AMP

3. 实验对比

### 8.2 V3

1. 多卡训练

## 9. 冒烟验证

每个训练后端版本引入时必须通过：

1. 最小数据集单 epoch 训练。
2. 日志采集验证。
3. best 权重产出验证。
4. pt 导出验证。
